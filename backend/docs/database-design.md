# 数据库设计说明

当前采用 **MySQL 8+**。

## 已确定的核心数据

### 1. 用户信息 `users`
用于存储：
- 用户账号
- 密码哈希
- 用户类型（管理员 / 普通用户）
- 昵称
- 头像
- 注册时间
- 最后登录时间
- 账号状态

### 2. 登录数据 `user_sessions`
用于存储：
- 登录 token
- refresh token（可选）
- 登录时间
- 过期时间
- 退出时间
- 登录 IP
- user-agent
- token 是否失效

### 3. 导航分组 `nav_groups`
用于存储：
- 某个用户自己的导航分组
- 分组名称
- 分组描述
- 排序
- 是否默认分组

### 4. 导航链接 `nav_links`
用于存储：
- 某个用户自己的导航网址
- 所属分组
- 网址标题
- 描述
- 本地模式链接
- 外网模式链接
- 图标地址
- 打开方式
- 排序

## 我补充的两张表

### 5. 用户设置 `user_settings`
这个我建议直接保留。

理由：你前端已经有这些概念了：
- 主题模式（夜间 / 白天 / 跟随系统）
- 跳转模式（本地 / 外网）
- 默认搜索引擎

这些都不适合硬塞到 users 表里，单独拆表更干净。

### 6. 操作日志 `operation_logs`
这个我也建议保留。

理由：后面你很可能会需要：
- 管理员改了谁的数据
- 用户什么时候删了网址
- 某次操作是否成功

先留一个日志表，后续非常省事。

## 关系说明

- 一个 `users` 可以有多条 `user_sessions`
- 一个 `users` 可以有多条 `nav_groups`
- 一个 `users` 可以有多条 `nav_links`
- 一个 `nav_groups` 可以有多条 `nav_links`
- 一个 `users` 只有一条 `user_settings`

## 安全建议

1. **密码绝对不要明文存储**
   - 数据库里只存 `password_hash`
   - 后端建议后面使用 `bcrypt`

2. **token 建议后续区分两种**
   - access token
   - refresh token

3. **删除导航数据建议优先软删除**
   - 现在 `nav_groups` / `nav_links` 都留了 `is_deleted`

## 当前阶段推荐顺序

1. 先执行 `sql/init.sql`
2. 验证数据库能连通
3. 做用户注册 / 登录
4. 做导航分组 CRUD
5. 做导航链接 CRUD
6. 最后把前端 localStorage 切换成真实接口
